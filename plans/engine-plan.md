## Architecture

### Approach: Event-Sourced State Machine with Middleware Guards

The Engine uses **event sourcing** as its core pattern: the state machine does not store mutable state. Instead, session state is derived by reducing the append-only event log. This directly satisfies Spec's requirement that "full event log → identical final state" and simplifies testing (replay any event sequence → verify state).

HTTP routes use **Hono middleware as state guards**: each route declares which states it accepts (e.g., `requireState("NEGOTIATING")`). Invalid-state requests are rejected at the middleware layer before reaching business logic, keeping handlers thin and secure.

### Data Flow

```
HTTP Request
    │
    ▼
Hono Middleware Stack
    ├── error-handler (catch + format JSON errors)
    │
    ▼
Route Handler (Spec-ordered validation)
    ├── 1. Calls Duty 1 schema validators (validateRfq, validateOffer, etc.)
    ├── 2. Field-level checks (price, currency, expiry) — per Spec order
    ├── 3. Authorization checks (from===buyer, signer identity)
    ├── 4. Ed25519 signature verification (verifyEd25519)
    ├── 5. ZK proof verification (verifyBudgetProof) — if applicable
    ├── 6. State machine guard (requireState check) — AFTER sig per Spec
    ├── 7. Appends event to EventStore
    │
    NOTE: Validation order follows Spec §8 exactly. Signature verification
    runs BEFORE state guard (per Spec). This prevents unauthenticated
    actors from probing session state via error code differences.
    │
    ▼
EventStore.append(event)
    ├── Stores event in ordered array
    ├── Notifies SSE subscribers (for GET /events)
    └── State is NOT mutated — derived on next read
    │
    ▼
Session state = reduce(events) → { state, offers, counters, quote }
```

### Concurrency Model

Each RFQ session is isolated in a `Map<rfq_id, Session>`. Requests to the same `rfq_id` are serialized using a per-session async queue (simple promise chain). This prevents race conditions (e.g., two counters arriving simultaneously) without global locks or external dependencies.

```typescript
// Conceptual per-session serialization
class SessionLock {
  private queue: Promise<void> = Promise.resolve()

  async run<T>(fn: () => Promise<T>): Promise<T> {
    let resolve: () => void
    const gate = new Promise<void>(r => { resolve = r })
    const prev = this.queue
    this.queue = gate
    await prev
    try {
      return await fn()
    } finally {
      resolve!()
    }
  }
}
```

### Security Architecture

**Read-path authorization (role-scoped views):**
- All read endpoints (`GET /events`, `GET /quote`) require authentication (Ed25519 signed request or bearer token)
- **Buyer** sees: full event stream (all offers, all counters, quote state) — this is the protocol's intended information advantage
- **Seller** sees: only their own offers, counters addressed to them, and quote state when they are the selected seller
- **Third party / unauthenticated**: no access to session internals (`401 Unauthorized`)
- Implementation: middleware extracts caller DID from request signature → `filterEventsForRole(events, callerDid, rfq)` before response

**Quote immutability and versioning:**
- After `POST /accept`, the unsigned quote is stored server-side with a `quote_revision` counter (starts at 1)
- `PUT /quote/sign` and `PUT /cosign` accept only `{ buyer_signature }` or `{ seller_signature }` — **no quote fields in the request body**
- Engine verifies the signature against the **stored quote's canonical JSON bytes**, not client-supplied data
- If a rollback occurs (seller decline, commitment timeout), the quote is invalidated. A new `POST /accept` creates a new quote with `quote_revision + 1`
- Any sign/cosign referencing an old revision → `409 stale_quote_revision`

**Event payload sanitization (anti-injection):**
- When appending events, extract ONLY typed known fields from the request body. NEVER store raw request body as `event.payload`.
- For offers: extract `{offer_id, rfq_id, seller, price, currency, valid_until, signature, extensions}` — nothing else
- For counters: extract `{counter_id, rfq_id, round, from, to, price, currency, valid_until, signature, budget_proof, extensions}` — nothing else
- Unknown top-level fields (e.g., attacker-injected `state`, `quote`) are silently dropped before storage
- `deriveState()` reducer reads ONLY from these typed fields — never destructures the full payload
- `extensions` are preserved per Spec §5.7 but stored separately from protocol fields to prevent namespace collision

**Minimum normalized amount validation:**
- After any price passes `isValidDecimalPositive()`, also check: `normalizeAmount(price, mint) > 0n`
- Prices that normalize to zero (e.g., `"0.0000001"` with USDC 6 decimals → 0 micro-units) are rejected with `422 invalid_amount`
- This check runs in: RFQ (anchor_price), Offer (price), Counter (price), and Accept (final_price derived from offer)
- Prevents zero-payment exploits where a valid-looking decimal truncates to zero on-chain

**Deadline re-check in sign and cosign:**
- `PUT /quote/sign` and `PUT /cosign` MUST verify `Date.now() < Date.parse(rfq.deadline)` in addition to `state === COMMIT_PENDING`
- Post-deadline sign/cosign → `409 invalid_state_transition` (same error as expired session)
- This closes a window where a fast buyer+seller pair could complete commitment after the global deadline

**Role-aware EventStore interface:**
- `EventStore.getEvents()` and `EventStore.subscribe()` accept a mandatory `callerDid` and `rfq` parameter
- Filtering happens INSIDE the interface, not at the application layer
- It is impossible to retrieve unfiltered events — the interface enforces role-scoping by construction
- SSE `subscribe()` callback only fires for events the caller is authorized to see

**SSE cleanup on session pruning:**
- When deadline enforcer prunes a terminal session, it MUST:
  1. Call all registered `unsubscribe()` functions for that session
  2. Send an SSE close event (`event: close\ndata: session_pruned\n\n`) to all connected clients
  3. Remove the session from the subscriber registry
- Prevents memory leak from orphaned SSE closures holding references to pruned sessions

**Counter verification order (Spec §8 compliant + DoS hardened):**
The 12-step counter verification follows **Spec §8 normative error code semantics** while adding a lightweight pre-check to prevent unauthenticated ZK CPU abuse:

**Pre-check (before Spec steps, ~0.1ms):**
- 0a. Verify `counter.signature` starts with `"ed25519:"` and base64 payload decodes to 64 bytes → `400 malformed_payload` if not
- 0b. Verify `counter.from` DID public key matches the signature's public key bytes (extract key from DID, compare with sig pubkey — NO full sig verification yet) → `400 malformed_payload` if mismatch
- This rejects garbage/forged requests before ANY expensive operation. Costs ~0.1ms (DID parse + byte compare), not 50ms (ZK) or 1ms (full sig verify)

**Spec §8 steps (normative order, correct error codes):**
1. Parse and validate CounterOffer schema → `400 malformed_payload`
2. Retrieve RFQ for `rfq_id` → `404`
3. Verify `counter.price` is valid positive decimal + `normalizeAmount(price, mint) > 0n` → `422 invalid_amount`
4. Verify `counter.currency` matches `rfq.currency` → `422 currency_mismatch`
5. Verify `counter.valid_until` is in the future → `422 invalid_expiry`
6. Verify `counter.from === rfq.buyer` → `422 unauthorized_counter`
7. ZK proof verification (if `rfq.budget_commitment` present):
   a. Check proof present → `422 missing_budget_proof`
   b. Resolve `mint = mintFor(rfq.currency)`, compute `expected_scaled = normalizeAmount(counter.price, mint)`
   c. Check `counter.budget_proof.counter_price_scaled === expected_scaled.toString()` → `422 proof_price_mismatch`
   d. Verify Groth16 proof (~50ms) → `422 invalid_budget_proof`
   If `rfq.budget_commitment` is absent and `counter.budget_proof` is present → `422 unexpected_budget_proof`
8. Validate buyer Ed25519 signature (full cryptographic verification) → `401 invalid_buyer_signature`
9. Check state === `NEGOTIATING` → `409 invalid_state_transition`
10. Validate round monotonicity → `422 invalid_round`
11. Append event to log
12. Return `201`

**Design rationale:** Spec §8 uses "MUST validate in this order" for error code semantics — clients expect `422 unauthorized_counter` before `401 invalid_buyer_signature`. We respect this. But adding a format-level pre-check (step 0) that rejects syntactically malformed signatures BEFORE the Spec sequence is not a Spec violation — it's an implementation-layer optimization that returns `400 malformed_payload` (which Spec step 1 already uses). Only requests with correctly-formatted signatures and matching DID reach the expensive ZK step. This eliminates the asymmetric DoS vector while preserving Spec-compliant error codes for legitimate requests.

**UUID uniqueness enforcement:**
- `rfq_id`, `offer_id`, `counter_id`, `quote_id` are enforced as globally unique within the engine's lifetime
- Engine maintains a `Set<string>` of all seen IDs. Duplicate → `409 duplicate_object_id`
- After session pruning, IDs are moved to a `tombstoneSet` that persists for the full retention window (60 minutes)
- **Tombstones are only created for requests that pass authentication** — unauthenticated requests with random UUIDs do NOT populate the tombstone set (prevents memory DoS via UUID flooding)
- On restart (in-memory MVP), tombstones are lost — accepted tradeoff. SQLite upgrade persists tombstones durably with UNIQUE constraints as the authoritative dedup layer.

**Offer admission control (anti-Sybil):**
- Per-session total offer cap: max 50 offers per session. Beyond → `422 session_offer_limit`
- Per-DID offer cap: max 5 offers per seller DID per session. Beyond → `422 seller_offer_limit`
- These caps prevent Sybil offer flooding (Chain-1 attack: 50 fake sellers × 10 offers = session kill)
- Offers count separately from the 500-event session cap. The event cap rejects non-terminal events; the offer cap specifically limits `OFFER_SUBMITTED` events from sellers.
- Optional hardening (production): require 8004 Agent Registry registration to submit offers. Non-registered sellers → `403 unregistered_seller`. This gates offer submission behind on-chain identity (with registration cost), making Sybil attacks economically expensive.

**`payment_endpoint` provenance (anti-redirection):**
- `buildUnsignedQuote()` sources `payment_endpoint` from the **original Listing** associated with the accepted seller, NOT from the offer body or registry enrichment at accept-time
- The Listing's `payment_endpoint` is recorded in the `OFFER_SUBMITTED` event when the seller first submits
- If the seller's Listing has no `payment_endpoint`, the accept fails with `422 missing_payment_endpoint`
- Client-side verification contract: buyer MUST verify `payment_endpoint` in the unsigned quote matches the seller's known endpoint before signing

**Global accept attempt limit:**
- Per-session: max 6 total accept attempts (across all sellers). Beyond → `422 accept_limit_exceeded`
- Combined with per-seller limit of 2, this prevents Sybil timeout cycling (5 Sybil sellers × 2 attempts = 10, capped at 6)

**Extension field preservation (Spec §5.7):**
- Engine MUST NOT strip `extensions` fields from protocol objects during relay
- When storing events, the full original protocol object (including any `extensions`) is preserved in the event payload
- `filterEventsForRole()` preserves extensions on forwarded events
- Engine MAY ignore unknown extensions for its own processing but MUST relay them intact

**Counter `to` validation:**
- Counter verification must check that `counter.to` is a seller DID that has submitted at least one offer for the RFQ → `422 unauthorized_counter` if not
- This is Spec §5.4 requirement, checked after `counter.from === rfq.buyer`

**Currency-to-mint resolution for ZK:**
- Before calling `normalizeAmount()` in ZK proof verification, the engine must resolve `rfq.currency` (e.g., `"USDC"`) to a mint address via the SPL Token Mint Table (Spec §9)
- `const mint = mintFor(rfq.currency)` → then `normalizeAmount(counter.price, mint)` → compare with `counter_price_scaled`

**Deadline check inside critical section:**
- When a request acquires the session lock, the handler's first action is: `if (Date.now() >= rfq.deadline) → reject 409 expired`
- This is checked using the **current wall clock at handler start** (not lock acquisition time), inside the lock
- Prevents a request that queued before deadline but runs after deadline from being accepted

**Signer identity verification (all write endpoints):**
- Every write endpoint verifies not just "is the signature valid?" but "does the signer match the expected role?"
- `POST /rfqs`: signer DID must match `rfq.buyer`
- `POST /rfqs/:id/offers`: signer DID must match `offer.seller`
- `POST /rfqs/:id/counter`: signer DID must match `rfq.buyer` (double-check against `counter.from`)
- `POST /rfqs/:id/accept`: signer DID must match `rfq.buyer` (Spec §5.6 step 2 — MUST be authenticated)
- `PUT /rfqs/:id/quote/sign`: engine verifies `buyer_signature` against `didToPublicKey(quote.buyer)` — not any valid key
- `PUT /rfqs/:id/cosign`: engine verifies `seller_signature` against `didToPublicKey(quote.seller)` — must be the selected seller

**Signed control envelopes (domain separation):**
- All state-changing actions (accept, decline, cancel, sign, cosign) require a signed control envelope, not just DID-based auth
- Envelope format: `{ envelope_id: "uuid-v4", action: "accept"|"decline"|"cancel", rfq_id: string, session_revision: string, payload: {...}, issued_at: ISO8601, expires_at: ISO8601, signature: "ed25519:..." }`
- `session_revision` is REQUIRED for ALL control actions (not just accept) — prevents replay of decline/cancel envelopes from a previous state
- `envelope_id` is a unique nonce (UUID v4) — tombstoned after first successful use. Replay of same envelope → `409 duplicate_control_envelope`. Prevents within-window replay attacks.
- Signature covers canonical JSON of the envelope (with `signature: ""`)
- Engine verifies: (1) signature valid for caller DID, (2) `action` matches the endpoint, (3) `rfq_id` matches the route param, (4) `issued_at` is recent (within 60s), (5) `expires_at` is in the future
- This prevents cross-session, cross-action, and cross-engine replay of captured requests
- For `accept` specifically, the payload MUST include `{ offer_id, session_revision }` — CAS (compare-and-swap) semantics

**Accept anti-griefing (commitment timeout abuse):**
- Per buyer-seller pair: max 2 accept attempts per RFQ session. Third attempt for the same seller → `422 accept_limit_exceeded`
- This prevents a buyer from repeatedly locking a seller via accept-then-timeout cycles
- Counter resets if the buyer selects a different seller
- Tracked in derived state via counting `WINNER_SELECTED` events per seller DID

**Client-side quote verification (documented contract):**
- Before signing, both buyer and seller MUST locally reconstruct the expected unsigned quote from the accepted offer and RFQ, and verify all fields match the engine-provided quote
- Specifically verify: `final_price`, `payment_endpoint`, `seller`/`buyer` DIDs, `service_type`, `spec_hash`, `expires_at`
- This is a **client responsibility** documented in the API contract, not enforced by the engine
- Defends against a compromised engine returning a tampered quote

**Known protocol-level information leaks (documented limitations):**
- **Counter price lower bound**: Rising counter prices across rounds reveal a tightening lower bound on `budget_hard`. ZK proofs prove `counter ≤ budget_hard` but don't hide the counter price itself. Mitigation: strategies should never counter above `budget_soft`.
- **SSE timing**: Event delivery timing reveals negotiation cadence (gaps indicate buyer is negotiating with other sellers). Mitigation: optional jitter in event delivery (not implemented in v1).
- **Buyer omniscient advantage**: Buyer sees all sellers' offers; each seller sees only their own. This is BY DESIGN (Spec §8 "Buyer drives the negotiation"). Seller implementers must be aware.

### Session Lifecycle Details

**Seller decline (co-sign refusal):**
- Seller reads the quote via `GET /quote`, decides not to sign
- Seller (or engine timeout) triggers `COMMIT_PENDING → NEGOTIATING` rollback
- Engine appends `COSIGN_DECLINED` event with `actor: seller_did`
- The pending quote is discarded (quote fields cleared from derived state)
- Buyer strategy can immediately re-select a different seller via `POST /accept`

**Commitment timeout (seller disappears):**
- If `COMMIT_PENDING` persists for longer than `commitment_timeout`, the deadline enforcer auto-reverts to `NEGOTIATING` and appends `COSIGN_TIMEOUT` event
- **Default: 60 seconds.** Engine-level config via `COSIGN_TIMEOUT_MS` env var, clamped to [15s, 120s]:
  - 60s default: comfortably covers reasoning models (Claude Opus/o1-pro 30-60s) + weak GPU inference + network latency
  - 15s minimum: prevents misconfiguration that breaks legitimate agents
  - 120s maximum: bounds worst-case Sybil grief to 120s × 2 attempts = 4 minutes per seller identity
  - Public/adversarial deployments: recommend 30-60s
  - Private/trusted/LLM-heavy deployments: recommend 90-120s
- This prevents a griefing attack where a seller accepts being selected but never cosigns, locking the buyer until the global deadline
- `commitment_timeout` is separate from `rfq.deadline` — it's a per-commitment window, not the session deadline
- The buyer can immediately re-select another seller after timeout
- Future: if Ghost Bazaar evolves into multi-tenant marketplace with varied seller profiles, per-RFQ timeout (option C) can be added in a protocol revision with engine-enforced min/max bounds

**Deadline race condition:**
- Deadline enforcer runs on interval. A request may arrive concurrently with expiry.
- Per-session lock serializes: the request acquires the lock first OR the enforcer does.
- If enforcer wins → session is `EXPIRED` → request gets `409 invalid_state_transition`
- If request wins → request processes normally → next enforcer tick sees deadline passed but state is already `COMMITTED` (or still active → expires then)
- Edge case: `POST /accept` arrives 1ms before deadline → accepted. The `expires_at` on the resulting quote provides a separate settlement time window.

**SSE cursor contract:**
- `GET /rfqs/:id/events` returns all events for the session
- `?after=<cursor>` returns only events after the cursor. Cursors are **opaque UUIDs** (not sequential integers) to prevent gap analysis that reveals competitor count (RT-H4). Internally events have monotonic IDs, but the external cursor is a mapped UUID.
- If `Accept: text/event-stream` header present → SSE mode: keep connection open, push new events as `data:` frames with `id:` set to `event_id`
- If no Accept header or `application/json` → JSON array mode: return current events and close
- Client reconnect: browser `EventSource` auto-sends `Last-Event-ID` header → engine resumes from that cursor
- Cleanup: on client disconnect, remove subscriber from EventStore listener set

**Session lock error handling:**
- Lock acquisition has a 5-second timeout. If exceeded → `503 Service Unavailable` with `error: "session_busy"`
- If the locked operation throws, the lock is released (finally block) and the error propagates as the appropriate 4xx/5xx
- Lock queue is bounded: max 10 pending requests per session. Excess → `429 Too Many Requests`

## Technical Considerations

- **Event Sourcing performance**: For MVP with in-memory storage and short-lived sessions (< 100 events), deriving state from events on every request is negligible cost. If session event counts grow large in production, add a snapshot mechanism (cache derived state, invalidate on new event).
- **SSE connection limits**: Node.js default HTTP keep-alive may limit concurrent SSE connections. For the demo (< 10 connections) this is irrelevant. For production, consider `HTTP/2` or connection pooling.
- **ZK verification latency**: `verifyBudgetProof` using snarkjs is ~50ms. This runs synchronously in the counter route handler. Acceptable for demo; monitor in production.
- **Deadline enforcer accuracy**: `setInterval` is not precise. For the demo, 1-second granularity is fine. For production, consider a priority queue sorted by deadline timestamp.
- **EventStore interface for future persistence**: The `EventStore` interface abstracts storage. MVP uses `InMemoryEventStore`. Future implementations can swap to `better-sqlite3` or PostgreSQL without changing route code.
- **Hono `@hono/node-server`**: Hono runs on multiple runtimes. For this project, use `@hono/node-server` to run on Node.js with full access to `crypto`, `@noble/ed25519`, and snarkjs.

## Non-Functional Requirements

### Rate Limiting & Request Bounds

- **Global**: Max 100 requests/minute per source IP (Hono middleware). Excess → `429 Too Many Requests`.
- **Per-DID**: Max 10 RFQ creations per minute per buyer DID. Prevents session flooding.
- **Per-session**: Already handled by session lock queue (max 10 pending → 429).
- **Request body size**: Max 64 KB per request (`Content-Length` check before JSON parsing). Oversized → `413 Payload Too Large`. This prevents memory exhaustion via large proof arrays or string fields.
- **Proof field cardinality**: `pi_a` max 3 elements, `pi_b` max 3x2, `pi_c` max 3 elements. Schema validation rejects oversized proof fields before any crypto processing.
- **MVP**: Simple in-memory counter with sliding window. Production: Redis or rate-limit middleware.

### Logging

- **Request logging**: Hono middleware logs `method`, `path`, `status`, `duration_ms` for every request. Use structured JSON format.
- **Event logging**: Every `EventStore.append()` logs the event type, rfq_id, and actor. Private state (budget_hard, floor_price, commitment_salt) MUST NEVER appear in logs.
- **Error logging**: All 4xx/5xx responses log the error code and request context. Stack traces logged for 5xx only.
- **MVP**: `console.log` with JSON format. Production: swap to `pino` via dependency injection.

### Metrics (MVP)

- `engine.sessions.active` — count of sessions not in terminal state (COMMITTED/EXPIRED/CANCELLED)
- `engine.events.total` — total events appended across all sessions
- `engine.requests.duration_ms` — per-route request latency
- MVP: expose via `GET /metrics` as JSON. Production: Prometheus export.

### Event Retention & Pruning

- **Active sessions**: all events retained in memory (append-only, no pruning)
- **Terminal sessions** (COMMITTED / EXPIRED / CANCELLED): retained for 60 minutes after terminal transition (agents may be async with significant latency; 5 minutes is too aggressive for production). Pruned by the deadline enforcer sweep.
- **Per-session event cap**: max 500 events per session. At cap → new non-terminal events (offers, counters) are rejected with `422 session_event_limit`. Terminal actions (accept, cosign, cancel) are still permitted to allow graceful conclusion. The session is NOT auto-expired — this prevents a griefing attack where one party intentionally floods events to kill a negotiation. Per-actor quota: max 100 events per DID per session.
- **Memory budget**: monitor via `EventStore.size()`. If total events across all sessions exceeds 10,000 (unlikely in demo), log a warning. No hard limit for MVP.
- **Production upgrade path**: `SqliteEventStore` persists events to disk; pruning becomes `DELETE WHERE settled_at < now() - retention_period`

### Degraded-Mode Behavior

| Dependency | Failure Mode | Engine Behavior |
|-----------|-------------|----------------|
| 8004 Agent Registry (`discoverAgent`) | RPC timeout / unavailable | `GET /listings` returns listings **without** `registry` field; log warning; no error to client. **All registry-returned data (name, uri, scores) is sanitized** — strip HTML, validate URL format, clamp scores to 0-100 range. Prevents SSRF/injection from malicious registry entries. |
| ZK verification (`verifyBudgetProof`) | snarkjs throws | `POST /counter` returns `422 invalid_budget_proof`; counter rejected; event not appended |
| SSE subscriber | Client disconnects mid-stream | Remove subscriber from listener set; no error; no resource leak |
| Deadline enforcer | `setInterval` drift | Worst case: session expires 1-2 seconds late. Acceptable for demo. Production: priority queue. |
